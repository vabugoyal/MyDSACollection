What was the earlier 9 step process and why was it so cumbersome?

      Why 9 Steps?

      The OLD process was designed for periodic batch updates (weekly/monthly), not daily refreshes:
      1. Steps 1-2: Generate cache for new feed data
      2. Step 3: Copy production to scratch (can't modify in-place)
      3. Steps 4-5: Delete old feed data (prepare for merge)
      4. Steps 6-7: Merge quotesidCache and unresolvedQuotesidCache
      5. Step 8: Merge securityCache + dedup (most expensive)
      6. Step 9: Atomic move to production

      Why So Slow?

      - Petabyte-scale data: /data/utd/ contains 10+ years × 252 dates/year × 100+ feeds
      - NFS latency: Network filesystem amplifies I/O costs
      - No incremental updates: Must process all dates even if only one changed
      - Deduplication: Full table scan required to find conflicts

      Why Consolidated Universe is Better?

      - Per-date granularity: Each date is independent, can regenerate any date without affecting others
      - Immutability: Once generated, directory never changes (no conflicts, no deduplication)
      - Parallelization: 252 dates × 50 grid workers = ~5 dates/worker (2-4 hours total)
      - Rich metadata: quoteSid, directFeedQuoteSid, gmcQuotesids, allQuotesids, options fields
      - Automated feed selection: 95% of SPNs have unambiguous feed selection, no manual intervention

  2-Minute Detailed Response

  Setup the Problem:
  "The original system was designed for periodic batch updates, not daily refreshes. It maintained a global security cache with first_seen_date and last_seen_date columns to
   track when each quotesid was available. The refresh process had 9 sequential steps:"

  1. Scan all historical dates for a feed to extract quotesids
  2. Compute first/last seen dates and lookup metadata from database
  3. Copy production cache to scratch directory (~500GB)
  4. Delete old feed data from copied tables
  5. Merge new data with cleaned tables
  6. Deduplicate conflicts
  7. Move merged tables back to production

  "This took 1-2 days for several reasons:"

  Key Bottlenecks:
  - Sequential dependencies: Step N+1 couldn't start until Step N completed
  - Global state: Any feed update required scanning entire history (6+ years of data) to recompute first/last seen dates
  - Heavy I/O: Copying and merging 500GB tables multiple times over NFS
  - NFS locking: Only one feed could refresh at a time to prevent race conditions
  - No incrementality: Even if only one date changed, we'd rebuild everything

  The Solution:
  "We redesigned the architecture around per-date immutability:"

  1. Per-date directories: /data/utd/consolidated_univ/{date}/ instead of global cache
  2. Independence: Each date is self-contained—no dependencies on other dates
  3. Parallelization: 252 trading days × 50 grid workers = ~5 dates per worker
  4. Write-once: Generate directly in final location, no copy/merge/move
  5. Automated feed selection: 4-rule hierarchy + 20 config overrides = 95% automation

  Results:
  - Runtime: 1-2 days → 2-4 hours (10-20x faster)
  - Scalability: 10+ trading desks can self-serve without understanding feed internals
  - Robustness: Failures are isolated—one date failing doesn't block others
  - No locks needed: Immutable writes eliminate race conditions

  Follow-Up Questions & Answers

  Q1: "Why couldn't you parallelize the old process?"

  "The old process had global state dependencies. Step 6 merged new quotesidCache data with existing data, requiring knowledge of what quotesids existed across ALL dates to
  compute first_seen_date and last_seen_date correctly. Additionally, Step 8's deduplication required scanning the entire merged table to find conflicts. With per-date
  directories, each date is independent—we don't need to know anything about other dates."

  Q2: "What was the biggest technical challenge?"

  "Feed selection automation. With 15+ feeds (IDC, BBG, ICE, CME, EUREX) providing data for the same securities, we needed deterministic rules. We designed a 4-rule
  hierarchy:"
  1. Prefer direct exchange feeds (ICE, CME) over consolidated feeds (IDC, BBG) for lower latency
  2. Use country-aware selection (e.g., ICE for US securities, BBG for Canadian)
  3. Resolve conflicts using trading desk priorities (s1trd > peeqtrd > jpeqtrd)
  4. Override with 20 explicit config rules for edge cases

  "This automated 95% of decisions while allowing manual overrides for the remaining 5%."

  Q3: "How did you validate correctness?"

  "We ran parallel production for 3 months:"
  1. Generated consolidated universe for date D
  2. Compared with existing security cache quotesids for same SPNs
  3. For discrepancies, investigated which feed was correct
  4. Used findings to refine the 20 config override rules
  5. Final validation: All 10+ trading desks reviewed their SPNs

  "After cutover, we monitored data quality alerts—quotesids with missing data despite last_seen_date indicating presence."

  Q4: "What would you do differently?"

  "With hindsight, I'd parallelize earlier. We initially generated per-date sequentially to validate correctness before parallelizing. In retrospect, we could've started
  with parallelization and used spot-checks for validation, cutting development time by 2-3 months. Also, I'd add more instrumentation upfront—we added logging/metrics
  reactively when debugging edge cases."

  Q5: "How does this scale to more data?"

  "Linearly. Each date is independent, so adding more dates just means more grid workers. We're currently at 252 dates × 50 workers = ~5 dates/worker. If we needed 10 years
  of history (2,520 dates), we'd use 500 workers for the same 2-4 hour runtime. The bottleneck is per-date processing (database lookups for options data), which we optimized
   with intermediate caching—252x reduction in API calls."
